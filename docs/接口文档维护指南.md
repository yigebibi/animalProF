## 文档更新原则

### 何时需要更新文档

- ✅ 新增接口时
- ✅ 修改接口参数时
- ✅ 修改接口响应格式时
- ✅ 删除接口时
- ✅ 修改接口认证方式时
- ✅ 修改错误码时

### 何时不需要更新文档

- ❌ 修改内部逻辑但不影响接口时
- ❌ 修复 bug 不影响接口行为时

## 更新步骤

### 步骤 1: 更新代码

修改 `backend/src/modules/` 下的相关文件：

- 控制器文件: `*.controller.ts`
- DTO 文件: `dto/*.dto.ts`
- 服务文件: `*.service.ts`

### 步骤 2: 更新 Swagger 注解

确保代码中的 Swagger 注解是最新的：

```typescript
// 更新接口描述
@ApiOperation({ summary: '接口说明' })

// 更新参数说明
@ApiProperty({ description: '参数说明' })
```

### 步骤 3: 更新 API接口文档.md

打开 `D:\selfProject\API接口文档.md`，更新相应部分：

1. 更新文档顶部的版本号和日期
2. 在对应模块中更新接口说明
3. 在 [变更日志] 部分记录本次变更
4. 如果是重大更新，更新文档版本号

### 步骤 4: 验证文档

运行以下命令验证文档：

```bash
cd backend

# 方式 1: 启动服务查看 Swagger
npm run start:dev
# 访问 http://localhost:3000/api/docs 确认

# 方式 2: 运行检查脚本
npm run docs:check
```

## 文档检查脚本

### 安装依赖

已包含在项目中，无需额外安装。

## 提交时检查

### Git Hooks (可选)

为了确保每次提交都检查文档更新，可以使用 git hooks。

在 `.git/hooks/pre-commit` 中添加：

```bash
#!/bin/sh
echo "🔍 检查 API 文档是否需要更新..."

# 检查是否有后端代码变更
BACKEND_CHANGES=$(git diff --cached --name-only | grep "^backend/" | wc -l)

if [ "$BACKEND_CHANGES" -gt 0 ]; then
  echo "⚠️  检测到后端代码变更"
  echo "   请确认已更新 API接口文档.md"
  read -p "   继续提交? (y/n) " -n 1 -r
  echo
  if [[ ! $REPLY =~ ^[Yy]$ ]]; then
    exit 1
  fi
fi
```

赋予执行权限：

```bash
chmod +x .git/hooks/pre-commit
```

## 版本号规范

文档版本号遵循语义化版本 (Semantic Versioning)：

```
MAJOR.MINOR.PATCH
```

- **MAJOR**: 不兼容的 API 变更
- **MINOR**: 向后兼容的功能新增
- **PATCH**: 向后兼容的问题修正

#### 版本号更新示例

- 添加新接口: `1.0.0` → `1.1.0`
- 修改接口参数: `1.1.0` → `1.2.0`
- 修复文档错别字: `1.2.0` → `1.2.1`
- 重大重构，不兼容旧版: `1.2.1` → `2.0.0`

## 常见问题

### Q: 文档和 Swagger 不一致怎么办？

**A**: Swagger 是根据代码注解自动生成的，是权威文档。以 Swagger 为准，同步更新 Markdown 文档。

### Q: 忘记更新文档怎么办？

**A**: 在下次提交时补充更新，并在变更日志中说明。

### Q: 如何快速定位需要更新的文档部分？

**A**: 使用文本编辑器的搜索功能，根据接口路径或模块名搜索。

### Q: 文档可以用中文还是英文？

**A**: 团队内部使用中文，接口文档保持中文。但代码中的注解建议使用英文，方便国际化。
